# slk 1.0-Readiness: Code-Quality, Test, Docs, and Reliability Spec

**Date:** 2026-05-03
**Status:** Draft (awaiting user review)
**Owner:** @gammons

## 1. Background

A code review of `slk` ahead of a public-launch push surfaced a small number of
high-leverage issues. The architecture and discipline are good (real
service-oriented layering, dependency-injection seams in place, fast green
test suite, low TODO count, thoughtful comments), but three files have
absorbed too much logic and a handful of reliability/contributor-experience
gaps remain. This spec captures the work to close those gaps before `slk`
gets meaningful public attention.

The headline findings the spec addresses:

- `internal/ui/app.go` is **5,160 LOC** with a **1,222-line `Update`** and
  a **462-line `View`**.
- `cmd/slk/main.go` is **2,077 LOC** with a **775-line `run()`**.
- `internal/ui/messages/model.go`, `internal/ui/thread/model.go`, and
  `internal/ui/sidebar/model.go` each contain `View`/`buildCache`
  functions in the 250-350 line range.
- `internal/ui` sits at **44.4%** coverage; `internal/slack` (the
  reliability-critical WebSocket/API client) at **54.5%**;
  `internal/ui/overlay` at **0%**; `cmd/slk` at **13.7%**.
- The README's project-size figures (9,300 LOC / 31 source / 24 test) are
  stale by ~3√ó. Real numbers: 28,547 LOC / 89 source files / 97 test files.
- No package-level `doc.go` files outside two packages. No
  `docs/ARCHITECTURE.md`, no `CONTRIBUTING.md`, no `AGENTS.md`, no
  documented code-quality standards.
- No panic-recovery boundary around the bubbletea program loop. No
  opt-in debug log path. User-reported crashes are currently
  un-actionable.

## 2. Goals

- Bring `slk` to 1.0 readiness on **testability**, **maintainability**,
  **reliability**, and **contributor-friendliness**.
- Decompose the three god-files and trim the worst-offending functions in
  sibling panel models to pragmatic size targets.
- Raise meaningful test coverage to a defensible per-package floor.
- Add the documentation a popular OSS Go project is expected to ship with.
- Add minimal reliability scaffolding (panic recovery + opt-in debug log)
  so user-reported crashes are actionable.
- Establish a written code-quality rubric (`AGENTS.md`) that future work
  is held against, so the same review doesn't have to be re-litigated.

## 3. Non-goals

- **No new user-visible features.** This is a quality / reliability spec.
- **No behavior changes.** Refactors must be observably identical to
  current behavior.
- **No CI/lint/Makefile tooling.** Items 11-13 from the review (CI
  coverage badges, `make coverage` target, `golangci-lint` config) are
  deferred to a follow-up spec.
- **No characterization-test scaffolding.** We rely on the existing
  test suite plus a manual smoke checklist; we don't write throwaway
  tests purely to lock in pre-refactor behavior.
- **No source-of-truth migration.** SQLite remains a cache; Slack
  remains authoritative.

on `main`:

## 5. Approach

Refactor first, then test, then doc, then reliability. Phasing is
sequential because each phase makes the next easier:

- The decomposition in Phase 1 produces the small testable surface that
  Phase 2 needs.
- The decomposed file structure in Phase 1 makes the `doc.go` files in
  Phase 3 actually informative.
- Phase 4's reliability scaffolding lives in `cmd/slk/main.go`, which
  Phase 1 has already broken into clean stages.

`AGENTS.md` is written **before** Phase 1 so every refactor and test
commit has a written rubric to be reviewed against.

## 6. Phased plan

### Phase 0 ‚Äî `AGENTS.md` (standards doc)

Write `AGENTS.md` at repo root. Sections:

- **Project layout** ‚Äî one-paragraph layer summary pointing at
  `docs/ARCHITECTURE.md` (Phase 3 will write that doc; the link can
  exist before the target does).
- **Function and file size** ‚Äî functions ‚â§120 lines; files ‚â§800 lines;
  `(*App).Update` ‚â§150 lines as a pure dispatcher. Exceptions
  documented inline with `//nolint:funlen // <reason>`.
- **Naming** ‚Äî Go conventions; no `Manager` / `Helper` / `Util` suffixes
  unless the type genuinely manages, helps, or is a utility namespace.
- **Comments** ‚Äî every exported identifier has a doc comment; non-obvious
  *why*-comments preferred over *what*-comments; package-level `doc.go`
  required for every package under `internal/` and `cmd/`.
- **Testing** ‚Äî test behavior, not call shape; `t.Parallel()` where
  safe; table-driven tests for branchy code; per-package coverage
  floors (75% non-trivial; 70% `internal/ui`; 75% `internal/slack`;
  ‚â•60% `internal/ui/overlay`; 40% `cmd/slk`); cargo-cult
  `assert.NotNil` tests are rejected in review.
- **Behavior preservation rules for refactors** ‚Äî existing tests must
  stay green at every commit; manual smoke checklist signed off at
  each phase boundary; no observable behavior change without a
  separate behavior PR.
- **Dependency-injection seams** ‚Äî clipboard, time, HTTP, filesystem,
  exec must be swappable for tests. New code follows the existing
  `clipboardReader` (`internal/ui/app.go:548`) pattern: typedef +
  package-level default var + `Set...` setter.
- **Error handling** ‚Äî wrap errors with `fmt.Errorf("...: %w", err)`;
  no silent `_ = err` discards; no `panic` in non-`main` packages
  except for true invariant violations (and document them).
- **Reliability** ‚Äî no `log.Fatal` in non-`main` packages; goroutines
  must have a documented termination path; locks documented with what
  they protect.
- **Smoke-test checklist** ‚Äî see ¬ß8 below; lives in `AGENTS.md` so
  contributors don't have to dig through specs.

Phase 0 exit: `AGENTS.md` committed; spec acceptance criteria reference
it as the source of truth.

### Phase 1 ‚Äî Decomposition (refactor only, no behavior change)

#### 1a. Split `internal/ui/app.go` (5,160 LOC)

Target file structure:

| File | Contents |
|---|---|
| `app.go` | type definitions (`App`, `Mode`, `Panel`, `View`, `editState`, message types), `NewApp`, `Init`, `SetClipboardReader`, `mixVersions`, panel-cache helpers |
| `app_update.go` | `(*App).Update` as a pure dispatcher ‚â§150 lines + per-`tea.Msg`-type methods (`onWindowSize`, `onKey`, `onMouse`, `onPreviewLoaded`, `onChannelSelected`, etc.) |
| `app_view.go` | `(*App).View`, layout helpers, panel-cache rendering |
| `app_keys_normal.go` | `handleNormalMode` and its sub-helpers |
| `app_keys_insert.go` | `handleInsertMode` and its sub-helpers |
| `app_keys_modal.go` | mode-specific key handlers (channel finder, workspace finder, theme switcher, presence menu, reaction picker) |
| `app_mouse.go` | mouse event handling currently inside `Update`'s switch |
| `app_async.go` | async-message handlers (preview load/cycle, fetch results, reconnect notifications) |

`(*App).Update` becomes a flat dispatcher ‚Äî one `switch` on `tea.Msg`
type, each arm calling exactly one method. No nested logic.

#### 1b. Split `cmd/slk/main.go::run()` (775 LOC)

Decompose into named stages:

- `parseFlags(args []string) (cliFlags, error)`
- `loadConfig() (config.Config, error)`
- `openCache(cfg config.Config) (*cache.DB, error)`
- `loadWorkspaces(ctx, db, cfg) ([]WorkspaceContext, error)`
- `bootstrapServices(ctx, workspaces) (*tea.Program, error)`
- `runProgram(p *tea.Program) error`
- `shutdown(workspaces []WorkspaceContext, db *cache.DB) error`

`main()` and `run()` become thin orchestrators (each ‚â§60 lines). New
files: `cmd/slk/run.go`, `cmd/slk/stages.go` as needed.

#### 1c. Trim oversized panel `View` functions

| Function | Current | Target |
|---|---|---|
| `internal/ui/messages/model.go::(*Model).View` | 352 | ‚â§120 |
| `internal/ui/thread/model.go::(*Model).View` | 328 | ‚â§120 |
| `internal/ui/sidebar/model.go::(*Model).buildCache` | 263 | ‚â§120 |

Approach: extract layout-math, style-selection, and per-section
rendering into private helpers. The messages package already has a
`render.go` ‚Äî relocate logic there.

**Phase 1 exit criteria:**

- All tests green.
- Smoke checklist (¬ß8) signed off.
- No file in the repo >800 LOC (or carries a documented exception).
- No function >120 LOC (or carries a documented exception).
- `(*App).Update` ‚â§150 LOC and is a pure dispatcher.

### Phase 2 ‚Äî Test backfill

Goal: hit the coverage floors in ¬ß4. Phase 1 has produced small
testable units; Phase 2 writes tests against them.

- **2a. `internal/ui` to ‚â•70%.** Unit-test each new `Update` per-msg-type
  method in isolation. Table-driven tests where the dispatch decisions
  are data-driven.
- **2b. `internal/slack` to ‚â•75%.** Add tests for:
  - Reconnection state machine (exponential backoff 1s ‚Üí 30s, jitter,
    cap, reset on success).
  - Event parsing edge cases: malformed JSON, unknown subtypes,
    missing fields, oversize payloads.
  - Auth failure paths: expired `xoxc`, rotated `d` cookie, 4xx on
    handshake.
- **2c. `internal/ui/overlay` to ‚â•60%.** Currently zero tests. Write
  unit tests for the overlay's lifecycle and rendering.
- **2d. Sweep packages currently <75% non-trivial up to floor:**
  `themeswitcher` (30.4%), `workspacefinder` (34.9%), `presencemenu`
  (41.6%), `reactionpicker` (42.4%), `imgrender` (46.8%), `service`
  (60.5%).
- **2e. `cmd/slk` to ‚â•40%.** Test the extracted stage functions from
  Phase 1b (config load, cache open, workspace load, flag parsing).

**Test-quality rules** (from `AGENTS.md`):

- Tests must exercise behavior, not call shape.
- `assert.NotNil(result)` after the SUT returns is not a test.
- Prefer one good integration-style test of the unit's public
  contract over many internal-state assertions.

**Phase 2 exit criterion:** `go test -short -cover ./...` shows every
package above its floor.

### Phase 3 ‚Äî Documentation

- **3d. `CONTRIBUTING.md`** at repo root. ~10 lines: how to build/test,

  for orientation, link to issue templates if any.

**Phase 3 exit criterion:** every internal package has a `doc.go`;
`docs/ARCHITECTURE.md` and `CONTRIBUTING.md` exist; README numbers are
accurate.

### Phase 4 ‚Äî Reliability

#### 4a. Panic-recovery boundary

Wrap `tea.Program.Run` in a `recover()` boundary. On panic:

1. Best-effort terminal restore (call program release hook, emit
   `\033c` reset to stderr if hook unavailable).
2. Write `~/.local/share/slk/crash-<unix-timestamp>.log` containing:
   - slk version and build commit
   - Go version, OS, architecture, terminal (`$TERM`,
     `$TERM_PROGRAM`)
   - Goroutine stack trace (`runtime/debug.Stack`)
   - Last N=64 events from a small in-memory ring buffer of
     `tea.Msg` *type names* (no payload bodies ‚Äî see privacy note
     below)
3. Print the crash-log path to stderr before exit.
4. Exit code 2.

**Privacy note.** The crash log captures stack traces and event
**type names**, never event payloads. Message text, channel names,
and tokens never enter the crash log. This is encoded as a unit
test on the ring buffer.

#### 4b. Opt-in debug log

Add `--debug-log` CLI flag. When set:

- Install a structured logger (stdlib `log/slog` with a JSON or text
  handler ‚Äî pick text for human-readability) writing to
  `~/.local/share/slk/debug.log`.
- Cap at 5MB. On overflow, rename to `debug.log.1` (overwriting any
  prior `.1`) and start fresh. Single rotation, no compression.
- Existing `log.Printf` calls (audit and migrate) route through it.
- Off by default. The flag is documented in `--help` and in the
  README "Reporting bugs" section added in 4c.

**Privacy note.** The debug log can contain message metadata (channel
IDs, timestamps, event types) but **must never** log message bodies,
token values, or cookies. A linter-style test scans
`internal/slack/auth.go` and similar token-handling files for
`log.` calls and fails if any are added.

#### 4c. Documentation

Add to README a "Reporting bugs" section pointing users at the crash
log and `--debug-log` flag. Document both in `docs/ARCHITECTURE.md`.

**Phase 4 exit criterion:** induced panic in dev produces a usable
crash log; `--debug-log` produces a usable debug log; both are
documented.

## 7. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Refactor in Phase 1 introduces a subtle behavior change in `Update` dispatch | Smoke checklist (¬ß8) signed off after each commit. Existing 97 test files cover most behavior. Phase 2's tests catch regressions retroactively. |
| Coverage chasing in Phase 2 produces low-value tests | `AGENTS.md` rule: tests exercise behavior, not call shape. Reviewer pushes back on cargo-cult tests. |
| `doc.go` files become stale | Keep them short (paragraph + key types). Stale prose still beats no prose. |
| Crash log accidentally logs secrets | Crash log captures stack + event *type names* only. Unit test verifies ring buffer never stores payloads. |
| Debug log accidentally logs tokens or message bodies | Audit `log.Printf` call sites during migration; regression test scans token-handling files for `log.` calls. |
| Worktree drift from `main` during multi-week work | Rebase weekly. Phase boundaries are clean merge points if the user wants to ship intermediate value. |
| Spec scope creep | Non-goals (¬ß3) are explicit. New asks become follow-up specs. |

## 8. Smoke-test checklist

Run after each phase boundary. Tester signs off in the merge PR.

Boot `./bin/slk` against a real workspace, then exercise:

- [ ] Open a channel from the sidebar.
- [ ] Send a plain message.
- [ ] Send a multi-line message (`Shift+Enter`).
- [ ] Edit own message (`E`), confirm.
- [ ] Delete own message (`D`), confirm.
- [ ] Add a reaction via picker (`r`).
- [ ] Toggle existing reaction (`R` then `Enter`).
- [ ] Switch workspaces via `1`, then `2`.
- [ ] Fuzzy-find a channel (`Ctrl+t`), select it.
- [ ] Open a thread (`Enter` on a parent), reply.
- [ ] Scroll messages up to backfill (`Ctrl+U` repeatedly).
- [ ] Mark a message unread (`U`), reopen channel, confirm landmark.
- [ ] Image preview (`O`) on an inline image; cycle (`h`/`l`); close.
- [ ] Theme switch (`Ctrl+y`).
- [ ] Status set (`Ctrl+s`) ‚Üí DND 20m ‚Üí unset.
- [ ] Quit cleanly (`q`, confirm).

Any failure halts the phase merge until fixed.
